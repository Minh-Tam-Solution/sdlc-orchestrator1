# ADR-040: App Builder OpenSpec Integration
## SDLC Framework 6.0 Specification Standard Integration

**Status**: APPROVED
**Date**: January 28, 2026
**Deciders**: CTO, Backend Lead, Architect
**Sprint**: Sprint 106 - App Builder Integration (MVP)
**Framework**: SDLC 6.0 (Section 8 - Unified Specification Standard)

---

## Context

### Problem Statement

App Builder generates code scaffolds without proper SDLC documentation. This creates a disconnect between:
- Generated code (what was built)
- Requirements specification (why it was built)
- Quality requirements (how it should be validated)

**Before OpenSpec Integration:**
```
Generated Project/
├── src/
├── package.json
├── README.md            # Basic documentation only
└── .gitignore
```

### Business Impact

1. **No Governance Compliance**: Generated projects don't follow SDLC Framework methodology
2. **Missing Audit Trail**: No specification document for evidence vault
3. **Tier Confusion**: Projects don't have clear quality requirements based on tier
4. **Manual Documentation**: Developers must manually create SDLC specs post-generation

---

## Decision

**Integrate SDLC Specification Standard v6.0 (OpenSpec) into App Builder.**

Every generated project will include:
1. **SPEC-{id}-{project}.md** - SDLC 6.0 compliant specification
2. **YAML Frontmatter** - Machine-parseable metadata
3. **BDD Requirements** - GIVEN-WHEN-THEN format
4. **Tier-Specific Quality Requirements** - LITE/STANDARD/PROFESSIONAL/ENTERPRISE

**After OpenSpec Integration:**
```
Generated Project/
├── docs/
│   └── SPEC-1001-my-app.md    # NEW: SDLC 6.0 Specification
├── src/
├── package.json
├── README.md
└── .gitignore
```

---

## Implementation

### 1. Schema Changes (TemplateBlueprint)

**New Enums:**
```python
class ProjectTier(str, Enum):
    LITE = "LITE"               # Small projects, minimal requirements
    STANDARD = "STANDARD"       # Typical projects, standard requirements
    PROFESSIONAL = "PROFESSIONAL"  # Regulated industries
    ENTERPRISE = "ENTERPRISE"   # Mission-critical systems

class SpecCategory(str, Enum):
    FUNCTIONAL = "functional"
    TECHNICAL = "technical"
    SECURITY = "security"
    PERFORMANCE = "performance"
    INTEGRATION = "integration"
```

**New Fields in TemplateBlueprint:**
| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `spec_id` | str | `SPEC-{uuid[:8]}` | Unique specification identifier |
| `tier` | ProjectTier | STANDARD | Project tier classification |
| `sdlc_stage` | str | "03" | Current SDLC stage (BUILD) |
| `spec_category` | SpecCategory | TECHNICAL | Specification category |
| `owner` | str | "Backend Team" | Specification owner |
| `description` | str | "" | Project description |

### 2. YAML Frontmatter Generation

**Method:** `TemplateBlueprint.get_openspec_frontmatter()`

```yaml

```

### 3. Specification Content Sections

**Method:** `BaseTemplate.generate_spec_file()`

Generated specification includes:

| Section | Content | Source |
|---------|---------|--------|
| 1. Overview | Purpose, scope, stakeholders | `blueprint.description`, `blueprint.owner` |
| 2. Requirements | BDD format (GIVEN-WHEN-THEN) | `blueprint.entities`, `blueprint.features` |
| 3. Technical | Tech stack, env vars, quality mode | `blueprint.tech_stack`, `blueprint.env_vars` |
| 4. Data Model | Entity definitions, relationships | `blueprint.entities` |
| 5. API Contracts | Route definitions | `blueprint.api_routes` |
| 6. User Interface | Page specifications | `blueprint.pages` |
| 7. Quality | Tier-specific requirements | `blueprint.tier` |

### 4. Tier-Specific Quality Requirements

| Tier | Documentation | Testing | Security | Performance |
|------|--------------|---------|----------|-------------|
| **LITE** | README only | Smoke tests | Basic validation | < 3s load |
| **STANDARD** | README + API docs | Unit (60%) + Integration | OWASP Top 10 | < 500ms API |
| **PROFESSIONAL** | Full specs + ADRs | 80% coverage + E2E | ASVS L1 | < 200ms API |
| **ENTERPRISE** | Complete package | 90% + Load tests | ASVS L2 + Pentest | < 100ms API |

### 5. Integration with Scaffold Workflow

**Updated `BaseTemplate.scaffold()` method:**

```python
def scaffold(self, blueprint: TemplateBlueprint) -> List[GeneratedFile]:
    # Step 1: Validate blueprint
    self.validate_blueprint(blueprint)

    # Step 2: Collect all files
    files: List[GeneratedFile] = []

    # Step 3: Generate SDLC 6.0 Specification (NEW)
    spec_file = self.generate_spec_file(blueprint)
    files.append(spec_file)

    # Step 4-8: Generate other files...
    files.extend(self.generate_base_files(blueprint))
    # ... rest of scaffold
```

## Consequences

### Positive

1. **Full SDLC Compliance**: Every generated project follows Framework 6.0
2. **Automatic Documentation**: No manual spec creation needed
3. **Quality Clarity**: Tier-specific requirements are explicit
4. **Evidence Ready**: Spec file can be stored in Evidence Vault
5. **AI-Parseable**: YAML frontmatter enables automated processing
6. **BDD Format**: Requirements are testable (GIVEN-WHEN-THEN)

### Negative

1. **Larger Output**: Each project gains 1 additional file (~200-500 lines)
2. **Generation Time**: Slight increase (~10ms for spec generation)
3. **Schema Complexity**: TemplateBlueprint has 6 new fields

### Neutral

1. **Learning Curve**: Developers must understand SDLC 6.0 spec format
2. **Maintenance**: Spec template may need updates with Framework changes

---

## Alternatives Considered

### Alternative 1: Reference Existing Specs

**Approach:** User provides existing SPEC-XXXX ID, App Builder reads and generates code to match.

**Why Rejected:**
- Requires spec to exist before generation
- Doesn't help new projects starting from scratch
- More complex workflow

### Alternative 2: Wait for OpenSpec Conversion Layer

**Approach:** Wait for Sprint 117 conversion layer that converts proposals to specs.

**Why Rejected:**
- Delays App Builder value delivery
- Conversion layer is for different use case (proposal → spec)
- App Builder has all data needed to generate spec directly

### Alternative 3: External Spec Generator

**Approach:** Generate code first, use separate tool to create spec.

**Why Rejected:**
- Two-step process increases friction
- Risk of spec-code mismatch
- Loses atomic generation benefit
