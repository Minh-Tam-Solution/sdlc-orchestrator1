# 💡 HMW (How Might We) Questions Worksheet

**Purpose**: Transform problems into opportunities through generative questions
**When to Use**: DEFINE → IDEATE transition (SDLC Stage 01-02 - WHAT? → HOW?)
**Time Required**: 30-60 minutes
**Participants**: Product team, designers, stakeholders

---

## 📋 How to Use This Worksheet

1. **Complete POV Statement first** - HMWs derive from POV insights
2. **Generate 20-50 HMWs** - Quantity over quality at this stage
3. **Use HMW formula** - "How might we [action]?" (not "How can we")
4. **Diverge first, converge later** - Generate all possibilities, then prioritize
5. **Use HMWs for ideation** - Each HMW becomes brainstorming prompt

---

## ✅ HMW QUESTION FORMULA

### **"How might we [action/opportunity]?"**

**Why "Might"**:
- ✅ Suggests possibility (not certainty)
- ✅ Invites multiple solutions (not one right answer)
- ✅ Reduces pressure (exploration, not commitment)

**Why NOT "Can"** or "Should"**:
- ❌ "How can we" suggests feasibility constraint (premature)
- ❌ "How should we" implies one right answer (limits exploration)

---

## 🎯 HMW GENERATION STRATEGIES

Use these 7 strategies to generate diverse HMWs from your POV:

---

### Strategy 1: Amp Up the Good 📈

**Turn positive aspects into opportunities**

**POV Element**: [Something that works well but could be better]
**HMW**: How might we [amplify/enhance/scale] this positive element?

**Example (NQH-Bot)**:
```
POV Element: Managers already trust security footage as verification
HMW: How might we make verification as trustworthy as security footage but instant (not 45 min)?
HMW: How might we capture the same visual proof automatically at check-in?
HMW: How might we give staff access to their own visual verification?
```

---

### Strategy 2: Remove the Bad 🗑️

**Eliminate pain points or obstacles**

**POV Element**: [Major pain point from journey map]
**HMW**: How might we [eliminate/remove/prevent] this pain point?

**Example (NQH-Bot)**:
```
POV Element: Manual data entry causes 5-10% error rate
HMW: How might we eliminate manual data entry entirely?
HMW: How might we prevent errors at the source (check-in point)?
HMW: How might we make data entry impossible (auto-capture)?
```

---

### Strategy 3: Explore the Opposite 🔄

**Flip the problem on its head**

**POV Element**: [Current approach or assumption]
**HMW**: How might we do the OPPOSITE of current approach?

**Example (NQH-Bot)**:
```
Current Approach: Manager verifies staff attendance
Opposite HMW: How might we make STAFF verify their own attendance?
Opposite HMW: How might we make CUSTOMERS verify staff presence (via service quality)?
Opposite HMW: How might we make verification UNNECESSARY (trust by default)?
```

---

### Strategy 4: Question an Assumption 🤔

**Challenge hidden beliefs**

**POV Element**: [Implicit assumption in current approach]
**HMW**: How might we [challenge assumption] and explore alternatives?

**Example (NQH-Bot)**:
```
Assumption: Attendance must be tracked daily
HMW: How might we track attendance by output (work completed) instead of hours?
HMW: How might we verify presence without tracking exact times?
HMW: How might we make attendance tracking irrelevant (task-based pay)?
```

---

### Strategy 5: Go After Adjectives ✨

**Turn negative adjectives into positive goals**

**POV Element**: [Negative adjective describing current state]
**HMW**: How might we make [opposite positive adjective]?

**Example (NQH-Bot)**:
```
Negative Adjective: Manual tracking is "tedious"
HMW: How might we make attendance tracking effortless?
HMW: How might we make check-in delightful (staff want to do it)?
HMW: How might we make verification instantaneous (not tedious)?
```

---

### Strategy 6: Identify Unexpected Resources 🎁

**Leverage existing assets in new ways**

**POV Element**: [Existing resource or capability]
**HMW**: How might we use [resource] in a new way to solve problem?

**Example (NQH-Bot)**:
```
Existing Resource: Staff all have smartphones
HMW: How might we use staff smartphones to capture attendance automatically?
HMW: How might we leverage GPS in smartphones for location verification?
HMW: How might we use phone cameras to provide visual proof?
```

---

### Strategy 7: Create an Analogy from Need or Context 🌐

**Borrow solutions from other domains**

**POV Element**: [User need]
**HMW**: How might we solve this like [industry X] solves [analogous problem]?

**Example (NQH-Bot)**:
```
Analogy: Banks verify identity with biometric + photo
HMW: How might we verify attendance like banks verify transactions (biometric + GPS + photo)?

Analogy: Uber tracks driver location in real-time
HMW: How might we track staff presence like Uber tracks drivers (real-time GPS)?

Analogy: Blockchain creates immutable audit trails
HMW: How might we create attendance records as immutable as blockchain?
```

## ✅ HMW QUALITY CHECKLIST

### Good HMWs are:

- [ ] **Possibility-focused**: "Might" (not "can" or "should")
- [ ] **Human-centered**: Focus on user benefit (not technology)
- [ ] **Broad enough**: Multiple solutions possible (not one right answer)
- [ ] **Narrow enough**: Actionable (not too abstract)
- [ ] **Positive framing**: Opportunity language (not problem language)

### Red Flags (Rewrite if true):

- ❌ **Solution embedded**: "How might we build a mobile app?"
- ❌ **Too narrow**: Only 1 solution possible
- ❌ **Too broad**: "How might we make things better?"
- ❌ **Technology-first**: "How might we use AI/blockchain?"
- ❌ **Negative framing**: "How might we stop X from happening?"

---

## 🧪 VALIDATION: Test Your HMWs

### Test 1: The Multiple Solutions Test
**For each HMW, can you brainstorm 10+ solutions in 5 minutes?**
- ✅ YES → HMW is generative, well-scoped
- ❌ NO → HMW too narrow or embedded solution, rewrite

---

### Test 2: The Positive Energy Test
**Read HMWs to team. Do they feel energized or constrained?**
- ✅ "Oh! We could try X, Y, Z!" → Good HMW
- ❌ "I guess we have to do X..." → Narrow HMW, rewrite

---

### Test 3: The POV Alignment Test
**Does each HMW trace back to POV insight?**
- ✅ YES → HMW addresses real user need
- ❌ NO → HMW drifted from user research, cut it

## 💡 HMW BRAINSTORMING SESSION TEMPLATE

**Use top 3-5 HMWs as prompts for ideation:**

### HMW Session Structure (60 minutes total):

**Warm-up (5 min)**:
- Review POV statement
- Review top HMW
- Set rules: No criticism, wild ideas encouraged, quantity over quality

**Ideation Round 1 (15 min)**:
- HMW: [Top HMW #1]
- Silent brainstorming: Everyone writes 10 ideas
- Share ideas: Each person reads their ideas aloud
- Target: 50+ ideas in 15 minutes

**Ideation Round 2 (15 min)**:
- HMW: [Top HMW #2]
- Repeat process
- Build on others' ideas
- Target: 50+ more ideas

**Ideation Round 3 (15 min)**:
- HMW: [Top HMW #3]
- Repeat process
- Combine ideas from previous rounds
- Target: 50+ more ideas

**Cluster & Prioritize (10 min)**:
- Group similar ideas into themes
- Vote on top 3 concepts
- Prepare for prototyping

**Debrief (5 min)**:
- What surprised us?
- Which ideas should we prototype first?
- Next steps

**Total Ideas Generated**: _____ (Target: 100-150 ideas from 3 HMWs)

## 💡 PRO TIPS

### Tip 1: Quantity Unlocks Quality
> "Generate 30-50 HMWs, not 5. The best insights come after the obvious ones are exhausted."

### Tip 2: Mix Perspectives
> "Invite people from different roles (eng, design, business, customer support) to generate HMWs. Diverse perspectives unlock breakthrough ideas."

### Tip 3: Revisit HMWs During Ideation
> "If ideation feels stuck, generate more HMWs. Sometimes the right question unlocks 100 ideas."

### Tip 4: HMWs Evolve
> "After prototyping and testing, you'll discover new insights. Generate new HMWs and iterate."
