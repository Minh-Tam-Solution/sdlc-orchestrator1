### The Stanford d.school 5-Phase Methodology

```
┌──────────────────────────────────────────────────────────────┐
│  1. EMPATHIZE 🧠                                            │
│     Deeply understand users' pain, context, needs           │
│     Output: User Persona + Pain Points + Journey Map       │
├──────────────────────────────────────────────────────────────┤
│  2. DEFINE 🎯                                               │
│     Frame the RIGHT problem to solve                        │
│     Output: Problem Statement + POV + HMW Questions        │
├──────────────────────────────────────────────────────────────┤
│  3. IDEATE 💡                                               │
│     Generate diverse solution possibilities                 │
│     Output: 50+ Ideas → Top 3 Concepts → Sketches         │
├──────────────────────────────────────────────────────────────┤
│  4. PROTOTYPE 🛠️                                            │
│     Build minimum testable version to learn                │
│     Output: Working Prototype + Test Plan                  │
├──────────────────────────────────────────────────────────────┤
│  5. TEST 🧪                                                 │
│     Validate with real users, iterate                       │
│     Output: User Feedback + Iteration Plan + Ship Decision │
└──────────────────────────────────────────────────────────────┘
```

**Integration with SDLC 4-Stage Framework**:
```yaml
Stage 00 (WHY?):
  - Design Thinking Phase 1: EMPATHIZE (user research)
  - Design Thinking Phase 2: DEFINE (problem framing)
  - Output: Validated problem worth solving

Stage 01 (WHAT?):
  - Design Thinking Phase 2: DEFINE (outcomes)
  - Design Thinking Phase 3: IDEATE (start)
  - Output: Solution concepts validated

Stage 02 (HOW?):
  - Design Thinking Phase 3: IDEATE (complete)
  - Design Thinking Phase 4: PROTOTYPE (design)
  - Output: Technical design + prototype plan

Stage 03 (BUILD):
  - Design Thinking Phase 4: PROTOTYPE (build)
  - Design Thinking Phase 5: TEST (validate)
  - Output: Validated, shipped feature
```

## 🔄 System Thinking Integration (Pillar 3 Enhanced)

### The Iceberg Model Connection

**4-Layer System Thinking** (existing in SDLC 4.7):
```
Layer 1: EVENTS (visible symptoms)
         ↕️
Layer 2: PATTERNS (recurring themes)
         ↕️
Layer 3: STRUCTURES (systems/processes)
         ↕️
Layer 4: MENTAL MODELS (beliefs/assumptions) ← DESIGN THINKING TARGETS THIS
```

**Design Thinking Empathy** connects directly to **Layer 4 (Mental Models)**:
- Traditional analysis stops at Layer 2-3 (patterns, processes)
- Design Thinking digs to Layer 4 (why users think/feel/behave this way)
- **Result**: Solve root causes, not just symptoms

**Example** (NQH-Bot):
```yaml
Layer 1 Event: "Staff don't log daily reports"
Layer 2 Pattern: "70% non-compliance for 3 months"
Layer 3 Structure: "Paper forms + manual data entry"
Layer 4 Mental Model (Design Thinking Insight):
  "Staff don't trust management will use data fairly"
  "Fear of being judged/penalized based on reports"

Solution (Layer 4):
  - Anonymous aggregation option
  - Positive reinforcement system
  - Transparent data usage policy
  Result: 17% → 83% daily usage (+388%)
```

**Without Design Thinking**: Would have optimized the form (Layer 3 fix)
**With Design Thinking**: Addressed trust issue (Layer 4 fix) → Sustainable behavior change
